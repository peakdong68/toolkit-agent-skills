# 结构化规划

## 概述

结构化规划旨在在编写任何代码之前，将模糊的需求转化为经过批准且已文档化的实施计划。它强制要求进行澄清提问、带有权衡分析的方案对比以及用户的明确批准——从而避免最常见的时间浪费原因：构建了错误的东西。无论任务看起来多么简单，都必须经过此流程。

**启动时宣告：**"我正在使用规划技能来制定结构化的实施计划。"

## 触发条件

- 用户请求新功能、功能增强或更改
- Bug 修复需要超过一行的代码更改
- 跨多个文件的重构工作
- 任何尚未记录方案并获批准的任务
- 从头脑风暴技能过渡，且设计已获批准
- 调用 `/plan` 命令

## 阶段 1：上下文收集

**目标：** 在提问之前，先理解代码库和现有模式。

1. 阅读相关文件、文档、近期提交记录以及 `CLAUDE.md`
2. 检查记忆文件，了解已知的项目上下文、技术栈和约定
3. **检查 `docs/changes/<date>_<topic>/specs/*.md` 中是否存在现有规范** —— 如果该功能存在规范文件，加载其验收标准、数据契约和边界情况；这些将成为阶段 4 中任务定义的权威来源
4. 查阅 `docs/changes/<date>_<topic>/plan.md` 中现有计划，了解相关或依赖的工作
5. 识别新工作应遵循的现有模式
6. 记录在探索过程中发现的技术约束

**暂停 —— 在以下条件满足前，切勿进入阶段 2：**

- [ ] 你已探索代码库的相关部分
- [ ] 你已理解现有架构和模式
- [ ] 你已检查记忆文件以了解先前的决策
- [ ] 你已加载 `docs/changes/<date>_<topic>/specs/*.md` 中的相关规范并提取了其验收标准

## 阶段 2：澄清问题

**目标：** 通过一次一个针对性提问来消除歧义。

1. 每条消息只问 **一个** 问题 —— 切勿用多个问题轰炸用户
2. 尽可能使用选择题形式
3. 先研究代码库再提问 —— 不要问你能自己发现的东西
4. 将模糊的需求转化为具体、可测试的标准

### 问题类别优先级

| 优先级 | 类别     | 示例问题                                          |
| ------ | -------- | ------------------------------------------------- |
| 1      | 目的     | "这解决了什么问题？为谁服务的？"                  |
| 2      | 成功标准 | "我们如何知道它起作用了？完成的标准是什么？"      |
| 3      | 约束条件 | "是否存在性能、兼容性或时间方面的约束？"          |
| 4      | 非目标   | "我们应该明确**不**构建什么？"                    |
| 5      | 现有模式 | "我们应该遵循 X 中使用的模式，还是需要新的方案？" |
| 6      | 边界情况 | "当 [边界条件] 发生时应该怎么办？"                |

### 问题规则

| 规则             | 理由                               |
| ---------------- | ---------------------------------- |
| 每条消息一个问题 | 防止认知过载                       |
| 优先选择题       | 回答更快，减少歧义                 |
| 提问前先研究     | 尊重用户时间 —— 能自己发现的就别问 |
| 可测试的标准     | 模糊的回答会导致模糊的实现         |

**暂停 —— 在以下条件满足前，切勿进入阶段 3：**

- [ ] 你已理解目的和成功标准
- [ ] 你已识别约束条件和非目标
- [ ] 不存在未解决的关键歧义

## 阶段 3：方案设计

**目标：** 提出 2-3 个具体方案，包含权衡分析，并给出明确的推荐。

每个方案应包含：

| 章节       | 内容                   |
| ---------- | ---------------------- |
| 架构概要   | 2-3 句话描述该方案     |
| 关键文件   | 需创建/修改的准确路径  |
| 依赖       | 外部依赖或破坏性变更   |
| 权衡       | 明确的优缺点           |
| 工作量估算 | 任务数量（而非小时数） |
| 风险级别   | 低 / 中 / 高，附说明   |

### 方案选择决策表

| 因素               | 权重   | 如何评估                   |
| ------------------ | ------ | -------------------------- |
| 与现有模式的一致性 | 高     | 是否与当前代码库约定匹配？ |
| 简洁性             | 高     | 满足需求的最少移动部件     |
| 可测试性           | 中     | 每个组件能否独立测试？     |
| 未来可扩展性       | 低     | 仅在用户提及未来计划时考虑 |
| 性能               | 视情况 | 仅在用户指定性能约束时考虑 |

**首先给出你的推荐方案。** 解释为什么在给定约束下它是最佳选择。展示替代方案以表明你已考虑权衡空间。

**暂停 —— 在以下条件满足前，切勿进入阶段 4：**

- [ ] 你已提出至少 2 个方案
- [ ] 每个方案均已记录权衡分析
- [ ] 你已给出明确的推荐并附推理

## 阶段 4：计划文档

**目标：** 编写详细、可执行的计划文档并获得明确批准。

### 计划文档格式

**[模板：`plan-template.md`]** —— 本技能目录下同目录的模板文件为权威计划结构。生成计划文档时加载它。以下总结各必填章节；完整标记语言参见模板文件。

**必填文档章节：**

| 章节                                 | 用途                                                                          | 必填             |
| ------------------------------------ | ----------------------------------------------------------------------------- | ---------------- |
| 标头（目标、技术栈、设计文档、前置） | 界定范围、技术栈和依赖                                                        | 始终             |
| 改动总览                             | 一段话概括 + 要点 + 关键决策                                                  | 始终             |
| Task 0：环境准备                     | 验证运行时、构建工具、测试框架                                                | 始终             |
| Task 1..N                            | 背景（场景/原因/影响）、文件、步骤（位置/操作/原因）、验证（命令 + 预期输出） | 每个工作单元一个 |
| 认知变更                             | 需记录至 CLAUDE.md 的新文件/导出/约定                                         | 适用时           |
| Checkpoint 总结                      | 已构建内容、剩余工作                                                          | 每 2-3 个 Task   |
| 最终验收                             | 端到端验证及故障排查                                                          | 最后 Task        |

### 计划质量检查清单

| 标准                                        | 检查项                            |
| ------------------------------------------- | --------------------------------- |
| 标头包含目标、技术栈、设计文档、前置        | 无缺失的标头字段                  |
| 改动总览用要点概括所有修改                  | 禁止模糊的"各种变更"              |
| Task 0 验证环境工具链（运行时、构建、测试） | 禁止"假设工具可用"                |
| 每个 Task 均有背景（场景、原因、影响）      | 禁止无目的的孤立任务              |
| 每个步骤均有位置、操作、原因                | 禁止无具体细节的"修复这个 Bug"    |
| 每个验证均有具体命令 + 预期输出             | 禁止"肉眼检查"或"测试通过"        |
| 存在规范时 Task 引用规范 AC                 | 禁止脱离规范锚点的孤立任务        |
| Task 按依赖关系排序                         | 禁止向前引用                      |
| 每个 Task 耗时 2-5 分钟                     | 禁止"实现整个模块"                |
| TDD 步骤明确（先测试，后代码）              | 每个 Task 遵循 RED-GREEN-REFACTOR |
| 每 2-3 个 Task 后出现 Checkpoint 总结       | 禁止 5 个以上连续无间隔 Task      |
| 最终验收 Task 包含 E2E 验证 + 故障排查      | 禁止"应该能正常工作"              |
| 新文件/导出/约定记录认知变更                | 禁止未记录的项目记忆              |

将计划保存至 `docs/changes/<date>_<topic>/plan.md`。

**暂停 —— 在以下条件满足前，切勿进入阶段 5：**

- [ ] 计划文档已编写并保存
- [ ] 每个任务均包含文件路径、步骤和验证方法
- [ ] 用户已明确批准计划（回复"是"、"批准"、"继续"等）

## 阶段 5：过渡到执行

**目标：** 将已批准的计划移交至相应的执行技能。

### 步骤 1：考虑工作树隔离

执行前，评估计划是否适合使用隔离的 git 工作树：

| 条件                                             | 建议                 |
| ------------------------------------------------ | -------------------- |
| 多任务计划，可能与当前工作分支冲突               | 创建工作树           |
| 需要长时间运行（测试、构建）同时继续其他开发工作 | 创建工作树           |
| 同时处理多个并行任务/分支                        | 创建工作树           |
| 单一独立任务，无其他进行中的工作                 | 跳过 —— 使用当前仓库 |

如果适合使用工作树，调用 `using-git-worktrees` 技能并进入创建的工作树后再继续。创建 worktree 通常是执行计划的第一步。

### 步骤 2：选择执行技能

| 场景                                        | 下一技能                      | 理由                 |
| ------------------------------------------- | ----------------------------- | -------------------- |
| 标准实施（少于 3 个任务）                   | `task-management`             | 顺序跟踪执行         |
| 大型实施（包含 3 个及以上可独立实现的任务） | `subagent-driven-development` | 并行执行，带审查关卡 |
| 自主开发会话                                | `autonomous-loop`             | Ralph 风格的迭代执行 |
| 单一聚焦任务                                | `executing-plans`             | 直接执行计划         |

调用所选技能并传入计划文档路径。

## 反模式 / 常见错误

| 反模式                 | 失败原因                       | 正确做法                     |
| ---------------------- | ------------------------------ | ---------------------------- |
| "这太简单了不需要规划" | 简单任务也有未经验证的假设     | 照常规划 —— 计划可以很短     |
| "我已经知道方案了"     | 你的方案可能与项目模式冲突     | 记录下来并获取批准           |
| "用户想要快点"         | 糟糕的代码比规划好的代码更耗时 | 规划能防止返工               |
| "我边做边规划"         | 那是即兴发挥，不是规划         | 先规划，后执行               |
| 一次性问 5 个问题      | 让用户不知所措，得到模糊回答   | 每条消息只问一个问题         |
| 只提出 1 个方案        | 缺乏权衡分析，可能错过更优选项 | 始终提出 2-3 个方案          |
| 模糊的文件引用         | "更新测试" —— 哪个测试？       | 始终使用准确文件路径         |
| 耗时 30 分钟以上的任务 | 太大，难以跟踪和验证           | 拆分为 2-5 分钟的任务        |
| 未经批准就开始写代码   | 若方向变更将浪费精力           | 等待明确的"同意"             |
| 跳过 Task 0            | 工具未验证，后续任务阻塞       | 始终先验证工具链             |
| 步骤模糊缺少位置       | "添加验证" —— 在哪里？         | 每个步骤均需位置、操作、原因 |
| 验证缺少预期输出       | "测试通过" —— 那是什么样子的？ | 具体命令 + 可验证的预期输出  |

## 反自我辩解防御机制

<HARD-GATE>
在以下条件满足前，**切勿**编写任何代码、创建任何文件或采取任何实施行动：
1. 你已提出澄清问题并理解了需求
2. 你已提出带有权衡分析的方案
3. 用户已明确批准该计划

此规则适用于**每一项**任务，无论其看起来多么简单。
</HARD-GATE>

**铁律：未经批准的计划，绝不写代码。** 没有例外。没有"就做这一件小事"。没有"这很明显"。

如果你脑海中出现以下任何想法，**立即停止**：

- "我就快速搞一下……" —— 不行。先规划。
- "这不需要完整的计划……" —— 需要。计划可以很简短。
- "我之后再补文档……" —— 不行。先写文档。

## 子代理调度机会

| 任务模式                   | 调度目标                                                          | 时机                         |
| -------------------------- | ----------------------------------------------------------------- | ---------------------------- |
| 规划期间的独立研究任务     | `Agent` 工具，`subagent_type="Explore"`                           | 从代码库多个区域收集上下文时 |
| 跨架构层的计划验证         | `Agent` 工具调度 `planner` 代理                                   | 当计划覆盖多个系统边界时     |
| 计划批准后，独立的实施任务 | `Agent` 工具（多个并行，遵循 `dispatching-parallel-agents` 技能） | 当计划步骤之间无依赖关系时   |

调度时请遵循 `dispatching-parallel-agents` 技能协议。

## 具体示例

### 示例：小型 Bug 修复计划

```markdown
# 修复：登录按钮禁用状态未清除 — Plan (1) — 单任务

**目标：** 修复登录失败后登录按钮仍保持禁用的问题
**技术栈：** React 18, TypeScript, Jest, React Testing Library
**架构：** LoginForm 组件中的状态管理 Bug —— isSubmitting 在错误时未重置
**方案：** 在 handleSubmit 的 catch 块中重置 `isSubmitting` 状态
**设计文档：** 无（Bug 修复，无规范）
**前置：** 无

## 改动总览

LoginForm.tsx 中单行修复：在错误 catch 块中重置 isSubmitting 状态。无 API 或 schema 变更。

- 修改: `src/components/LoginForm.tsx` —— 在 catch 块中添加 setIsSubmitting(false)
- 关键决策：catch 块修复 vs. finally 块重构 —— catch 块是最小变更，风险更低

---

### Task 0: 环境准备

**背景：**
验证 React/TypeScript/Jest 工具链可用。

**执行步骤：**

- [ ] 验证 Node.js 运行时
  - `node --version`
  - 预期: v18.x 或更高
- [ ] 验证 npm 可用
  - `npm --version`
  - 预期: 版本号
- [ ] 验证测试框架
  - `npx jest --version`
  - 预期: 版本号

**检查步骤：**

- [ ] 构建成功
  - `npm run build`
  - 预期: 无错误
- [ ] 现有测试通过
  - `npm test`
  - 预期: 全部测试通过

---

### Task 1: 编写按钮重新启用的失败测试

**背景：**

- 场景：登录表单在提交时禁用按钮，但在错误响应后未能重新启用
- 原因：当前无测试覆盖失败后的禁用→启用状态转换
- 影响：Task 2 修复的前置条件；无下游依赖

**涉及文件：**

- 修改: `tests/components/LoginForm.test.tsx`

**执行步骤：**

- [ ] 添加登录失败后按钮重新启用的测试用例
  - 位置: `tests/components/LoginForm.test.tsx` —— 在现有 LoginForm describe 块内
  - 操作: 添加一个测试，mock 失败的 API 响应，模拟表单提交，断言错误后提交按钮未禁用
  - 原因: 测试必须先失败（RED 阶段）以验证修复是必要的

**验证检查点：**

- [ ] 新测试失败（RED）
  - `npm test -- --grep "re-enables button after failed login"`
  - 预期: 1 个失败测试

---

### Task 2: 修复 Bug

**背景：**

- 场景：LoginForm 的 handleSubmit 函数
- 原因：isSubmitting 在 API 调用前设为 true，但在错误时从未重置为 false
- 影响：依赖 Task 1 的测试；无下游任务

**涉及文件：**

- 修改: `src/components/LoginForm.tsx`

**执行步骤：**

- [ ] 在错误 catch 块中重置 isSubmitting
  - 位置: `src/components/LoginForm.tsx` —— handleSubmit 函数，catch 块
  - 操作: 在 catch 块的第一行添加 `setIsSubmitting(false)`
  - 原因: finally 块方案也能实现但修改更多代码；catch 块是 Bug 修复的最小变更

**验证检查点：**

- [ ] 所有 LoginForm 测试通过（GREEN）
  - `npm test -- --grep "LoginForm"`
  - 预期: 全部测试通过，包括 Task 1 新增的测试
- [ ] 完整测试套件通过
  - `npm test`
  - 预期: 无回归

**认知变更：**

- [ ] [CLAUDE.md] LoginForm 的 isSubmitting 状态必须在成功和错误两条路径上都重置
```

## 验证关卡

在声明计划完成前，请验证：

1. **识别 (IDENTIFY)：** `docs/changes/<date>_<topic>/plan.md` 中存在计划文档
2. **运行 (RUN)：** 根据质量检查清单审查计划的完整性
3. **阅读 (READ)：** 验证所有章节均已填写具体细节
4. **规范核对 (SPEC-CHECK)：** 如果 `docs/changes/<date>_<topic>/specs/*.md` 中存在规范，验证每个任务至少引用了一项验收标准
5. **验证 (VERIFY)：** 用户已明确批准
6. **声明 (CLAIM)：** 仅在满足以上条件后，才能过渡到实施阶段

---

## 核心原则

- **DRY** —— 不要重复自己
- **YAGNI** —— 不要构建当前不需要的东西
- **TDD** —— 适用时先写测试
- **频繁提交** —— 每个任务完成后进行小型、原子化提交
- **准确路径** —— 计划中始终指定准确的文件路径

## 技能类型

**严格 (RIGID)** —— 严格遵循此流程执行每项实施任务。各阶段按顺序进行且不可妥协。未经批准的计划，绝不编写代码。
