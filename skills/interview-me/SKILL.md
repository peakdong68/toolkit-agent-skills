# 访谈我（Interview Me）

## 概述

人们所要求的与他们真正想要的往往是两回事。他们要求"一个仪表盘"，是因为这就是人们通常会要求的东西，而不是因为仪表盘能解决他们的问题。他们说"让它更快"，却没有给出具体要达到的数字指标。

发现这种差距成本最低的时刻，是在任何计划、规格或代码存在之前。一旦你开始构建，转换成本就变得真实存在，用户会将错误的东西合理化为"足够好"的东西。不匹配的问题就会被锁定。

这项技能能在产生任何成本之前弥合差距。其他定义阶段的技能假设你已经大致知道想要什么：`brainstorming`（头脑风暴）从一个想法生成变体，`spec-writing`（规格撰写）将需求写下来。而 interview-me 是所有这些之前的环节，你每次只问一个问题，附上你的最佳猜测，直到你能在用户说出来之前就预测出他们要说什么。

## 使用时机

在以下情况下应用此技能：

- 需求缺少以下至少一项：**谁**是用户、**为什么**他们想要它、**成功**是什么样子、约束**条件**是什么
- 请求是常规性的而非具体明确的（"给我构建X"、"让它更快"），且你无法在不猜测的情况下拆解这个常规说法
- 你忍不住想要从尚未明确表达的假设开始
- 用户没有说明当两个合理的价值目标存在冲突时，他们优先优化哪一个（简洁性 vs. 灵活性，成本 vs. 速度）
- 用户明确调用："访谈我"、"拷问我"、"开始之前，我们确定吗？"、"压力测试我的想法"

**何时不使用：**

- 需求明确且自包含（"重命名这个变量"、"修复这个拼写错误"）
- 用户已明确要求速度优先于验证
- 纯信息请求（"X是如何工作的？"、"这段代码做什么？"）
- 机械性操作（重命名、格式化、文件移动）
- 你已经拥有≥95%的置信度；在假设自己未达到之前，请重新阅读下方的停止条件

## 加载约束

此技能需要一个实时、可响应的用户。**请勿在非交互式上下文中调用**，如CI流水线、定时任务、`/loop` 或 `autonomous-loop`。如果你处于这些环境中且需求描述不充分，请将其标记为阻碍用户的问题，而非进行猜测。

## 流程

### 步骤1：提出假设，附带置信度数值

在询问任何问题之前，用**一句话**写下你当前对用户想要什么的最佳理解，并附上一个诚实的置信度数值（0–100%）：

```
假设：你想要一种在站会上回答"我们进展如何？"的方式，而"仪表盘"是你脑海中浮现的常规答案。
置信度：~30% — 缺失项：为谁服务、上下文中"指标"的含义、成功是什么样子
```

这个数值迫使你说实话。如果你写下一个高数值，却无法真正预测用户对接下来三个问题的反应，那么这个数值就是错的。从你能捍卫的置信度水平开始。

当置信度低于约70%时，在同一行附加一个简短的原因说明——还有什么未解决或缺失。这能准确告诉用户访谈需要挖掘什么，并防止这个数值成为一个模糊的信号。

### 步骤2：每次只问一个问题，每个问题都附带猜测

格式：

```
问：<一个聚焦的问题>
猜测：<你对答案的假设，以及产生该假设的推理>
```

等待用户回应后再问下一个问题。

**为什么一次只问一个，而不是一批：**

- 如果你把假设埋在列表里，用户就无法对你的假设做出反应
- 批量提问鼓励略读和表面化的回答
- 第三个问题往往取决于第一个问题的答案；一次性问所有问题会锁定错误的框架
- 用户认真思考的精力是有限的；一次一个问题地消耗它

**为什么附带猜测：**

- 用户对错误猜测的反应速度比从头生成答案更快
- 它让你承诺一个可能被明显证明错误的假设，这能保持你的诚实
- 它暴露**你的**假设，而这正是访谈旨在揭示的内容

这里的风险是礼貌的用户为了迎合而同意你的猜测。缓解方法是明显表现出愿意犯错的态度，并偶尔朝你预期用户会反驳的方向进行猜测。

### 步骤3：倾听"想要"与"应该想要"的区别

最危险的答案是用户说出听起来像深思熟虑的回答，而非他们真正想要的内容。留意以下情况：

- 答案模式匹配最佳实践话术（"我想要它可扩展"、"干净的架构"）但缺乏具体细节
- 答案诉诸惯例（"大多数应用的做法"、"标准方法"）
- 诸如"我可能应该…"、"我想我被要求…"、"良好的工程实践说…"之类的短语
- 将流行词作为目标——当"现代化"、"可扩展"、"健壮"成为答案本身，而非具体成果时

当你听到这些时，要问的问题是：

> _"如果你不必向任何人证明这个决定的合理性，你真正想要的是什么？"_

这一个问题往往比之前五个问题加起来更有效。

### 步骤4：用用户自己的话重述意图

当你的置信度很高时，写回你现在认为用户想要的内容。保持简洁（5–8行），尽可能使用用户的语言，并采用结构化格式，让用户可以逐行确认或纠正：

```
以下是我现在认为你想要的：

- 成果：      <一行>
- 用户：      <一行 — 谁受益>
- 为何现在：  <一行 — 什么发生了变化>
- 成功标准：  <一行 — 我们如何知道它奏效了>
- 约束条件：  <一行 — 关键限制>
- 超出范围：  <一行 — 我们明确不做的事情>

确认 / 否认 / 细化？
```

包含"超出范围"是不可或缺的。一半的对齐失败源于对**不**构建什么内容的隐性分歧。

### 步骤5：确认——明确的"是"，而非"你觉得就行"

门槛是一个明确的"是"。以下情况**不算**"是"：

- "你觉得怎么最好就怎么做。" → 用户在委托决策，这意味着他们也没有95%的置信度。用两个具体选项重新提问，框定为选择。
- "听起来不错。" → 模糊。追问："有什么你想细化的吗？"沉默不是确认。
- "好的，开始吧。" → 通常是礼貌的退出，而非认可。同样追问。
- 沉默后跟着"好的我们开始吧。" → 用户已放弃访谈，而非达成共识。停下来问是否遗漏了什么。

如果他们纠正你，将修正内容纳入并重新陈述。循环直到获得明确的"是"。

### 步骤6：引导至下游技能——绝不直接跳到代码

**[HARD-GATE:ROUTE]** 在用户以明确的"是"确认重述之后，你必须推荐适当的下游技能。interview-me 是一个定义阶段的技能；它的工作是澄清意图，而非执行。已确认的意图作为下一阶段的输入——它永远不是实现之前的最后一步。

**推荐哪个下游技能：**

| 意图特征                                             | 引导至                       | 原因                                        |
| ---------------------------------------------------- | ---------------------------- | ------------------------------------------- |
| 具体明确：清晰的谁、什么、为什么、成功标准、约束条件 | `/spec-writing` 然后 `/plan` | 从已确认的意图撰写规格，然后规划实现        |
| 需要界定范围："我想要X但不知道它应该是什么形态"      | `/brainstorming`             | 基于现在明确的意图生成变体，然后重新检查    |
| 琐碎 / 自包含的操作（重命名、修复拼写）              | 无——直接继续                 | interview-me 本不应被调用；参见"何时不使用" |

**明确地格式化交接。** 确认后，最后输出一次重述，然后引导：

```
已确认。以下是总结：

- 成果：      <一行>
- 用户：      <一行>
- 为何现在：  <一行>
- 成功标准：  <一行>
- 约束条件：  <一行>
- 超出范围：  <一行>

下一步：我建议使用 /spec-writing 将其捕获为正式规格，然后用 /plan 制定实现计划。可以吗？
```

**确认后绝不能做的事：**

- 开始写代码或建议类库、框架、文件结构
- 产出任务列表、架构草图或组件树
- 生成除重述和引导推荐之外的任何产物
- 问"要我直接开始编码吗？"——你已经知道答案是否定的；引导即可

如果用户回退说"直接开始编码"，提醒他们：访谈挖掘出了他们想要什么，但尚未明确如何构建它。规格和计划的存在正是为了防止以正确的方式构建错误的东西。

### 95%置信度停止条件

当你能对以下问题回答"是"时，你就完成了：

> _我能否预测用户对接下来我会问的三个问题的反应？_

如果是，你们已达成共同理解。停止访谈并产出重述。如果否，你还未完成；问下一个问题。

这是一个可检验的测试，而非凭感觉。它也有一个底线：如果你已经进行了多轮问答仍无法预测，这本身就是关于需求的信息，而非继续硬撑的理由。停下来告诉用户："我已经问了X个问题，但仍无法预测你的反应。某些基础性的东西缺失了。要不要退一步重新思考？"

## 输出

此技能的产出是一份**已确认的意图陈述配上下游引导推荐**：来自步骤4的重述、来自步骤5的明确确认，以及来自步骤6的引导推荐。重述 + 引导 = 交付物。代码、任务列表和实现是下游产物；它们消耗此技能产出的意图——**通过被引导的技能间接访问，绝不直接进行**。

如果用户希望意图持久化（多会话项目、移交给其他协作者），可提供将其保存到 `docs/changes/<date>_<topic>/intent.md` 的选项。仅在用户确认后保存。

## 示例

一个简短的前后对比。

**不使用 `interview-me`：**

```
用户：给我构建一个我们指标的仪表盘
智能体：<开始提议图表库和布局方案>
```

智能体已假设：为谁服务、什么指标、"为我们团队"意味着什么、成功是什么样子。这些都没有明确说明。每个假设都是用户日后可以说"那不是我的意思"的地方。

**使用 `interview-me`：**

```
用户：给我构建一个我们指标的仪表盘

智能体：
假设：你想要一种在站会上回答"我们进展如何？"的方式，而"仪表盘"是你脑海中浮现的常规答案。
置信度：~30% — 缺失项：为谁服务、"指标"是什么、成功是什么样子

问：    当你说"我们进展如何？"时，是谁在问——仅你自己、站会上的工程团队，还是向上汇报（例如你的经理、高管、董事会）？
猜测：工程团队在站会上，因为"我们"通常指代这个范围，且站会是提出这个问题的场合。如果是给高管的，指标和表述方式会有很大不同。

用户：其实是给我自己的。我总是搞不清哪些实验正在运行，以及它们的早期信号看起来如何。

智能体：
更新理解：根本不是团队仪表盘。而是一个个人实验追踪器。置信度：~60% — 仍缺失："早期信号"的含义以及完成的标准。

问：    差距在于你不知道有哪些实验存在，还是无法在一个地方看到它们的结果？
猜测：第二个。你在某处有一个实验列表，但它们的结果分布在五个不同的工具中，手动整合这些结果让你感到疲惫。

用户：其实是第一个。我根本没有列表。它们分散在不同的文档里。

智能体：<继续访谈，达到95%，用户确认>

以下是我现在认为你想要的：

- 成果：      一个包含所有活跃实验及其基本状态的单一列表
- 用户：      你本人，追踪你自己的实验
- 为何现在：  实验分散在不同文档中；你无法一次性看到全部
- 成功标准：  打开一个页面就能看到每个实验及其当前阶段
- 约束条件：  必须从现有文档中提取；无需新的数据录入流程
- 超出范围：  数据分析、团队仪表盘、指标可视化、报告生成

已确认。

下一步：我建议使用 /spec-writing 将其转化为正式规格，然后用 /plan 来构建。可以吗？
用户：是
智能体：<以已确认的意图为输入，调用 spec-writing>
```

两个问题之后，智能体发现实际需求不是"一个仪表盘"，而是"一个列表"。不同的产物、不同的范围、不同的工作量。仪表盘方案本来是错误的。而且确认之后，智能体引导至 `/spec-writing` 而非直接跳到代码。

## 与其他技能的交互

**[HARD-GATE]** interview-me 是定义阶段的入口。确认后，你必须引导至一个下游技能。绝不直接跳到实现。

| 下游技能             | 何时引导                       | 关系                                          |
| -------------------- | ------------------------------ | --------------------------------------------- |
| `brainstorming`      | 意图需要界定范围或探索替代方案 | 生成变体，然后重新检查意图                    |
| `spec-writing`       | 意图具体明确且有清晰的成功标准 | 将意图正式化为带 Given/When/Then 的 JTBD 规格 |
| `planning`           | 规格存在之后                   | 从规格制定结构化的实现计划                    |
| `task-decomposition` | 计划存在之后                   | 将计划拆分为可执行的子任务                    |

**反模式：跳过链条。** 正确的链条是：

```
interview-me → spec-writing → planning → task-decomposition → 实现
                 ↑（如需界定范围，使用 brainstorming）
```

跳过任何一步都有返工风险。interview-me 的工作止于引导——它不产出规格、计划或代码。

## 常见合理化借口

| 合理化借口                               | 现实                                                                                                                           |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| "需求已经足够清晰"                       | 如果你现在无法用一句话写出用户期望的成果，那么需求就不清晰。在决定之前先执行步骤1。                                            |
| "问太多问题会浪费他们的时间"             | 4–6个针对性问题所浪费的时间很少。构建错误东西所浪费的时间是巨大的，而用户才是承担这一成本的人。                                |
| "我可以在构建过程中弄明白"               | 代码存在后的转换成本是现在的10倍。实现过程中的发现意味着返工。                                                                 |
| "他们说'你觉得就行'，所以我应该直接决定" | "你觉得就行"是委托，而非决策。用两个具体选项重新提问，框定为选择。                                                             |
| "我应该给他们几个选项来选择"             | 当用户知道自己想要什么并在权衡取舍之间选择时，选项才有效。他们还不知道自己想要什么。列出选项会扩大搜索范围；提问则会缩小范围。 |
| "如果我附带猜测，就是在引导他们"         | 引导正是目的所在。对猜测做出反应比从头生成答案更快。风险是阿谀奉承，而非引导；通过明显表现出愿意犯错来缓解。                   |
| "我们已经谈得够多了，我明白了"           | 测试一下：你能预测他们对接下来三个问题的反应吗？如果不能，说明你还没真正理解。                                                 |
| "用户说了是，我们完成了"                 | 如果"是"是在模糊的重述或开放式的"听起来不错"之后给出的，那么这个"是"是空洞的。具体重述并重新确认。                             |
| "用户确认了重述，我可以开始构建了"       | 确认不是编码的绿灯。它是下一个定义阶段技能的绿灯。引导至 `/spec-writing` 或 `/brainstorming`——绝不要跳转到实现。               |
| "意图很简单，不需要规格"                 | 简单的意图仍然可以从规格撰写中受益，以挖掘你可能没考虑到的边界情况和验收标准。如果确实微不足道，interview-me 本不应被调用。    |

## 危险信号

- 单条消息中包含三个或更多问题：这是批量提问，而非访谈
- 问题没有附带你的假设：这是调查，而非承诺
- 接受"你觉得怎么最好就怎么做"作为最终答案
- 在用户明确确认你的重述之前就产出规格、计划或任务列表
- 问题被框定为"什么才是最佳实践？"而非"你真正想要什么？"
- 用户给出彰显专业水准的答案（"可扩展"、"干净"、"现代化"），而你未深入探究这是否真是他们想要的就接受了
- 三轮或更多轮问答后你的置信度仍未明显提升：你在问错误的问题，退一步重新框架化
- 置信度数值低于约70%却未附带原因：如果用户不知道缺失什么，就无法帮助弥合差距
- 在用户确认之前就保存意图文档（文档本身暗示了用户并未给出的确认）
- 在重述中跳过"超出范围"一行（对非目标的隐性分歧是一半对齐失败的原因）
- 确认后不经过向下游技能的引导就直接跳到代码、类库建议或文件脚手架——这是成功访谈后最常见的失败模式
- 确认后问"要我直接开始编码吗？"或"准备构建了吗？"——你已经知道答案；引导即可

## 验证

应用 interview-me 后：

- [ ] 首轮对话中陈述了附带置信度数值的明确假设
- [ ] 每个低于约70%的置信度数值都附带了一行原因说明（什么仍待解决或缺失）
- [ ] 问题一次只问一个，每个问题都附带智能体的猜测
- [ ] 当用户给出彰显专业水准或惯例信号的答案时，至少运行了一次"如果你不必证明合理性，你真正想要什么？"的探询
- [ ] 向用户写回了具体的重述（成果 / 用户 / 为何现在 / 成功标准 / 约束条件 / 超出范围）
- [ ] 用户用明确的"是"确认了重述（而非"你觉得就行"、"听起来不错"或沉默）
