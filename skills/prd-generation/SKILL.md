# PRD 生成

## 概述

通过引导式探索，将高层级想法转化为结构化的产品需求文档。此技能引导完成问题/解决方案/约束条件的探索，生成包含可衡量目标和用户故事的专业 PRD，并在保存前确保利益相关者批准。

**开始时声明：** "我正在使用 prd-generation 技能来创建产品需求文档。"

## 第一阶段：探索

**一次只问一个问题**（尽可能提供多选）。

**不要跳过探索阶段**。即使用户提供了详细的简报，也请通过至少 3 个澄清性问题来确认理解。

探索结束后暂停——在起草前，先呈现收集到的答案摘要并获取确认。

### 问题空间问题

| #   | 问题                                 | 为何重要            |
| --- | ------------------------------------ | ------------------- |
| 1   | 这解决了什么问题？                   | 锚定整个 PRD 的核心 |
| 2   | 目标用户是谁？（角色画像、职能角色） | 塑造用户故事        |
| 3   | 用户目前如何解决这个问题？           | 识别竞争格局        |
| 4   | 不解决这个问题的影响是什么？         | 证明优先级合理性    |

### 解决方案空间问题

| #   | 问题                                   | 为何重要           |
| --- | -------------------------------------- | ------------------ |
| 5   | 成功是什么样子？（具体指标）           | 定义成功指标       |
| 6   | 哪些是必备功能，哪些是锦上添花的功能？ | 设定优先级层级     |
| 7   | 明确的非目标是什么？                   | 防止范围蔓延       |
| 8   | 是否有现有解决方案可供借鉴？           | 为设计决策提供信息 |

### 约束条件问题

| #   | 问题                                   | 为何重要         |
| --- | -------------------------------------- | ---------------- |
| 9   | 时间线是什么？是否有任何硬性截止日期？ | 界定发布计划范围 |
| 10  | 存在哪些技术约束？                     | 缩小解决方案空间 |
| 11  | 有哪些可用资源？                       | 设定现实期望     |
| 12  | 是否有合规性或监管要求？               | 识别非功能性需求 |

## 第二阶段：起草 PRD

使用下方模板生成 PRD。将收集到的答案分发给 `prd-writer` 智能体进行重度内容生成。

起草完成后暂停——在完成第三阶段审查之前，**不要**将其作为最终版本呈现。

### PRD 模板

```markdown
# [产品/功能名称] — 产品需求文档

## 1. 概述

用一段话总结这是什么以及为何重要。

## 2. 问题陈述

- 当前状况
- 痛点
- 不解决的影响

## 3. 目标与非目标

### 目标

- [ ] 目标 1（可衡量的）
- [ ] 目标 2（可衡量的）

### 非目标

- 明确不做 X
- 明确不做 Y

## 4. 用户故事

作为一个 [角色画像]，我想要 [执行操作]，以便 [获得收益]。

## 5. 功能需求

### FR-1: [需求名称]

- 描述
- 验收标准
- 优先级（P0/P1/P2）

## 6. 非功能需求

- 性能：[具体目标]
- 安全：[要求]
- 可访问性：[标准]
- 可扩展性：[期望]

## 7. 技术约束

- 平台/技术栈要求
- 集成依赖项
- 数据要求

## 8. 成功指标

| 指标 | 当前值 | 目标值 | 如何衡量 |
| ---- | ------ | ------ | -------- |

## 9. 时间线与里程碑

| 阶段 | 描述 | 目标日期 |
| ---- | ---- | -------- |

## 10. 待决问题

- [ ] 问题 1
- [ ] 问题 2

## 11. 附录

参考资料、线框图、相关文档
```

### 优先级分类

| 优先级 | 含义                 | 规则                   |
| ------ | -------------------- | ---------------------- |
| **P0** | 发布必备             | 没有这个，产品无法发布 |
| **P1** | 重要，发布后尽快交付 | 具有显著价值但非阻塞项 |
| **P2** | 锦上添花             | 提升体验，可以等待     |

## 第三阶段：审查

逐节呈现 PRD：

1. 每节结束后询问："这是否准确表达了您的意图？是否需要修改？"
2. 根据反馈在进入下一节之前进行修订
3. 特别关注以下高信号价值部分：
   - 目标与非目标（范围对齐）
   - 用户故事（角色画像准确性）
   - 成功指标（可衡量性）
   - 功能需求（验收标准完整性）

审查结束后暂停——在保存前获取明确的"批准"确认。

## 第四阶段：保存与过渡

获得明确批准后：

1. 保存至 `docs/prds/YYYY-MM-DD-<feature>.md`
2. 提交 PRD，提交信息为：`docs(prd): add PRD for <feature>`
3. 如果后续需要实施，调用 `brainstorming` 技能
4. 如果需要规格说明，调用 `spec-writing` 技能

### 过渡决策表

| 用户意图         | 下一技能                     | 理由                    |
| ---------------- | ---------------------------- | ----------------------- |
| "我们来构建这个" | `brainstorming` → `planning` | 先探索方法再规划        |
| "编写规格说明"   | `spec-writing`               | 将 PRD 拆解为 JTBD 规格 |
| "只需保存"       | 无                           | PRD 即为交付物          |
| "获取估算"       | `task-decomposition`         | 拆解为可估算的任务      |

## 反模式 / 常见错误

| 错误                        | 为何错误                     | 正确做法                               |
| --------------------------- | ---------------------------- | -------------------------------------- |
| 跳过探索直接起草            | 产生基于假设的 PRD           | 始终先完成第一阶段                     |
| 目标没有指标                | 无法衡量成功                 | 每个目标都需要具体数字                 |
| 缺少非目标                  | 必然导致范围蔓延             | 明确列出范围外内容                     |
| 用户故事缺少验收标准        | 需求不可测试                 | 为每个故事添加 Given/When/Then         |
| 泛泛的成功指标（"提升 UX"） | 不可衡量                     | 使用具体数字："将加载时间减少至 <2 秒" |
| 一次性呈现完整 PRD 供审查   | 用户难以消化，给出表面性批准 | 逐节呈现                               |
| 逐字复制竞品功能            | 忽略真实用户需求             | 聚焦用户问题，而非解决方案             |

## 反合理化防护

- **不要**因为"用户已经描述得足够清楚"而跳过探索
- **不要**在任何部分留置占位文本——填写每个部分或标注为"待定：[原因]"
- **不要**在未获得用户对每部分的明确批准前继续保存
- **不要**编造成功指标——它们必须来自用户

## 集成点

| 技能                  | 关系                                               |
| --------------------- | -------------------------------------------------- |
| `brainstorming`       | 上游：PRD 之前探索想法；下游：PRD 之后探索实施方案 |
| `spec-writing`        | 下游：PRD 提供高层级需求；规格说明用 JTBD 细化它们 |
| `planning`            | 下游：规划参考 PRD 需求进行任务拆解                |
| `task-decomposition`  | 下游：将 PRD 拆解为可估算的工作项                  |
| `tech-docs-generator` | 并行：PRD 告知需要哪些文档                         |
| `acceptance-testing`  | 下游：PRD 中的验收标准用于测试定义                 |

## 验证门禁

在声明 PRD 完成之前：

1. 验证所有 11 个部分均已填写（非占位文本）
2. 验证每个目标都有可衡量的指标
3. 验证非目标明确且具有实际意义
4. 验证用户故事包含验收标准
5. 验证用户已逐节批准
6. 验证文件已保存并提交

## 具体示例：探索摘要

```
问题：用户无法在仪表板中找到相关的搜索结果。
用户：数据分析师（主要），团队负责人（次要）。
当前变通方案：导出到 Excel 并使用 Ctrl+F。
不解决的影响：每位分析师每天浪费 30 分钟（12 人团队）。
成功指标：将平均搜索时间从 5 分钟减少至 <30 秒。
必备功能：跨所有仪表板组件的全文搜索。
非目标：高级布尔查询语法（P2，非发布范围）。
时间线：6 周完成 MVP。
约束：必须与现有 Elasticsearch 集群兼容。
```

在第二阶段开始之前，将此摘要呈现给用户确认。

## 技能类型

**灵活型** — 在保持"先探索后起草"原则和逐节审查流程的前提下，根据项目上下文调整探索深度和 PRD 结构。
