# 代码审查

## 概述

对照原始计划、编码规范及已学习的项目模式进行全面的代码审查。该技能会调度一个专用的代码审查智能体（agent）进行深入分析，确保每次更改在合并到主分支前都有据可依、与计划对齐且符合规范惯例。

**开始时声明：**“我正在使用代码审查技能来审查实现代码。”

---

## 第一阶段：收集上下文

**目标：** 明确哪些内容发生了更改、计划的要求是什么，以及适用哪些规范。

### 操作步骤

1. 获取待审查的更改：

```bash
git diff HEAD~N..HEAD          # 或指定提交范围
git log --oneline HEAD~N..HEAD # 完成了什么
```

2. 定位计划文档：

```bash
ls docs/plans/*.md | tail -1
```

3. **检查 `docs/specs/` 中是否存在现有规范** —— 如果该功能有对应的规范目录，请加载其验收标准、数据契约和边界情况；这些是代码实现必须满足的权威性行为预期

```bash
ls docs/specs/
```

4. 从 `memory/learned-patterns.md` 加载项目规范

5. 确认：
   - 更改了哪些文件
   - 计划/规范的要求是什么
   - 适用哪些规范

### 停止 — 在满足以下条件前，请勿进入第二阶段：

- [ ] 所有已更改的文件均已确认
- [ ] 计划文档已加载（如适用）
- [ ] 规范验收标准已加载（如存在规范）
- [ ] 已从记忆中加载相关规范
- [ ] 你能明确说明原本计划构建的内容

---

## 第二阶段：调度代码审查智能体

**目标：** 向 `code-reviewer` 智能体发送结构化的审查请求。

### 审查提示词模板

```
对照以下内容审查以下更改：
1. 计划：[计划文档或需求]
2. 规范：[规范文件路径，例如 `docs/specs/YYYY-MM-DD-<topic>/` —— 验收标准、数据契约、边界情况]
3. 规范惯例：[从记忆中学习的模式]
4. 标准：[CLAUDE.md 规则]

更改内容：
[git diff 输出或文件列表]

检查项：
- 规范符合度（实现是否满足每一项验收标准？）
- 计划对齐度（我们构建的是否与规格说明一致？）
- 代码质量（DRY、YAGNI、命名、结构）
- 错误处理（边界情况、故障模式）
- 安全性（注入、XSS、认证问题）
- 测试覆盖率（更改是否经过测试？测试是否对应规范的验收标准？）
- 性能（明显的瓶颈）
- 文档（公共 API 是否有文档说明？）
```

### 停止 — 在满足以下条件前，请勿进入第三阶段：

- [ ] 审查请求已发送
- [ ] 审查智能体已返回审查结果

---

## 第三阶段：分类与解决问题

**目标：** 对发现的问题进行分类，并修复所有“关键（Critical）”级别的问题。

### 问题分类表

| 类别                    | 定义                                               | 所需操作               |
| ----------------------- | -------------------------------------------------- | ---------------------- |
| **关键（Critical）**    | 缺陷、安全问题、数据丢失风险、违反计划、不符合规范 | 合并前必须修复         |
| **重要（Important）**   | 代码质量、缺少测试、违反规范惯例                   | 合并前应当修复         |
| **建议（Suggestions）** | 样式、命名、次要改进                               | 锦上添花，有时间则修复 |

### 修复循环

针对“关键”和“重要”级别的问题：

1. 修复问题
2. 运行测试以验证修复
3. 针对该修复重新调度代码审查智能体
4. 重复此过程，直到不再存在“关键”级别问题

### 停止 — 在满足以下条件前，请勿进入第四阶段：

- [ ] 所有“关键”级别问题已解决
- [ ] 所有“重要”级别问题已解决，或已明确给出理由并推迟处理
- [ ] 所有修复后测试套件全部通过

---

## 第四阶段：自我学习集成

**目标：** 将审查过程中发现的模式持久化保存，供后续会话使用。

### 操作步骤

1. 如果识别出新模式，更新 `memory/learned-patterns.md`
2. 如果发现常见错误，记录下来供未来参考
3. 如果计划需要调整，更新 `memory/decisions-log.md`

---

## 审查输出格式

```markdown
## 代码审查摘要

**范围：**[已审查的文件/组件]
**计划对齐度：**[对齐 / 轻微偏差 / 重大偏差]
**规范符合度：**[符合 / 部分符合 / 不符合 —— 按验收标准逐项评估]

### 关键问题 (N)

1. **[问题标题]** — `file:line`
   问题：[描述]
   修复建议：[具体建议]

### 重要问题 (N)

1. **[问题标题]** — `file:line`
   问题：[描述]
   修复建议：[具体建议]

### 规范验收标准覆盖情况

| 验收标准编号 | 描述           | 覆盖代码/测试    | 状态                       |
| ------------ | -------------- | ---------------- | -------------------------- |
| 1            | [验收标准摘要] | [测试或代码引用] | [已覆盖 / 缺失 / 部分覆盖] |

### 表现良好的方面

- [正面观察]
```

## 决策表：审查深度

| 更改类型           | 审查深度                                               | 审查者               |
| ------------------ | ------------------------------------------------------ | -------------------- |
| 新功能 (>100 行)   | 全面审查：规范符合度 + 计划对齐度 + 质量 + 安全 + 测试 | code-reviewer 智能体 |
| 缺陷修复 (<50 行)  | 聚焦审查：回归测试 + 根本原因 + 修复正确性             | code-reviewer 智能体 |
| 重构（无行为变更） | 行为保持：所有测试通过 + 无回归                        | code-reviewer 智能体 |
| 配置/基础设施更改  | 安全 + 正确性：未暴露密钥、语法有效                    | code-reviewer 智能体 |
| 仅文档更改         | 准确性 + 完整性：与当前代码行为一致                    | 内联审查             |

## 反模式 / 常见错误

| 反模式                     | 错误原因                     | 正确做法                         |
| -------------------------- | ---------------------------- | -------------------------------- |
| 跳过“小修复”的审查         | 微小更改可能导致生产环境故障 | 审查所有更改                     |
| 无计划或规范进行审查       | 没有需求无法验证正确性       | 始终先加载计划和规范             |
| 修复问题后未重新运行测试   | 修复可能引入新缺陷           | 每次修复后运行完整测试套件       |
| 笼统的反馈（“看起来不错”） | 缺乏可操作性，遗漏真实问题   | 引用具体代码行并提供修复建议     |
| 仅自己审查自己的代码       | 作者盲区会遗漏缺陷           | 始终调度代码审查智能体           |
| 推迟处理关键问题           | 关键问题会演变为生产事故     | 合并前必须修复，绝无例外         |
| 忽略规范验收标准           | 实现可能遗漏必需行为         | 在审查中逐项明确检查每个验收标准 |

## 合理化借口 —— 如果你产生以下想法，请立即停止

| 借口                   | 现实                               |
| ---------------------- | ---------------------------------- |
| “这只是改个错别字”     | 错别字修复可能破坏 API。请审查。   |
| “我对这段代码很有信心” | 信心不等于正确性。请审查。         |
| “测试都通过了”         | 测试可能会遗漏缺陷。请审查。       |
| “只是样式/格式调整”    | 样式更改可能引入缺陷。请审查。     |
| “没人会注意到的”       | 这正是缺陷上线的时候。请审查。     |
| “我稍后再审查”         | “稍后”永远不会到来。现在就审查。   |
| “工期很紧”             | 发布缺陷的成本远高于审查。请审查。 |

## 子智能体调度机会

| 任务模式              | 调度至                                              | 适用场景                            |
| --------------------- | --------------------------------------------------- | ----------------------------------- |
| 审查多个独立文件/模块 | `Agent` 工具（`subagent_type="Explore"`）           | 审查范围跨越多个不相关的模块时      |
| 聚焦安全的审查        | 调用 `security-review` 技能的 `Agent` 工具          | 更改涉及认证、输入处理或外部 API 时 |
| 性能影响评估          | 调用 `performance-optimization` 技能的 `Agent` 工具 | 更改影响热路径或数据密集型操作时    |

调度时请遵循 `dispatching-parallel-agents` 技能协议。

## 集成点

| 技能                             | 关系                                       |
| -------------------------------- | ------------------------------------------ |
| `spec-writing`                   | 审查验证实现是否符合规范验收标准           |
| `planning`                       | 审查检查实现是否与已批准的计划一致         |
| `test-driven-development`        | 审查验证测试覆盖率及 TDD 合规性            |
| `verification-before-completion` | 审查是验证的前置条件                       |
| `self-learning`                  | 审查结果将输入到已学习的模式中             |
| `acceptance-testing`             | 审查检查是否所有标准都有对应的验收测试     |
| `systematic-debugging`           | 如果审查发现缺陷，则切换至调试技能         |
| `security-review`                | 审查过程中的安全发现将触发更深入的安全分析 |
| `auto-improvement` | 上游 —— 跟踪审查有效性 | 审查发现需要模式分析时 |

## 技能类型

**严格（RIGID）** —— 四阶段流程为强制要求。每次更改必须由 code-reviewer 智能体进行审查。未经审查不得合并。绝无例外。
