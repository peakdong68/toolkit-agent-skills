# 结构化规划

## 概述

结构化规划旨在在编写任何代码之前，将模糊的需求转化为经过批准且已文档化的实施计划。它强制要求进行澄清提问、带有权衡分析的方案对比以及用户的明确批准——从而避免最常见的时间浪费原因：构建了错误的东西。无论任务看起来多么简单，都必须经过此流程。

**启动时宣告：**“我正在使用规划技能来制定结构化的实施计划。”

## 触发条件

- 用户请求新功能、功能增强或更改
- Bug 修复需要超过一行的代码更改
- 跨多个文件的重构工作
- 任何尚未记录方案并获批准的任务
- 从头脑风暴技能过渡，且设计已获批准
- 调用 `/plan` 命令

---

## 阶段 1：上下文收集

**目标：** 在提问之前，先理解代码库和现有模式。

1. 阅读相关文件、文档、近期提交记录以及 `CLAUDE.md`
2. 检查记忆文件，了解已知的项目上下文、技术栈和约定
3. **检查 `docs/specs/` 中是否存在现有规范** —— 如果该功能存在规范文件，加载其验收标准、数据契约和边界情况；这些将成为阶段 4 中任务定义的权威来源
4. 查阅 `docs/plans/` 中现有计划，了解相关或依赖的工作
5. 识别新工作应遵循的现有模式
6. 记录在探索过程中发现的技术约束

**暂停 —— 在以下条件满足前，切勿进入阶段 2：**

- [ ] 你已探索代码库的相关部分
- [ ] 你已理解现有架构和模式
- [ ] 你已检查记忆文件以了解先前的决策
- [ ] 你已加载 `docs/specs/` 中的相关规范并提取了其验收标准

---

## 阶段 2：澄清问题

**目标：** 通过一次一个针对性提问来消除歧义。

1. 每条消息只问 **一个** 问题 —— 切勿用多个问题轰炸用户
2. 尽可能使用选择题形式
3. 提问前先研究代码库 —— 不要问你自己能发现的问题
4. 将模糊的需求转化为具体、可测试的标准

### 问题类别优先级

| 优先级 | 类别     | 示例问题                                            |
| ------ | -------- | --------------------------------------------------- |
| 1      | 目的     | “这解决了什么问题？目标用户是谁？”                  |
| 2      | 成功标准 | “我们如何知道它已生效？‘完成’的具体表现是什么？”    |
| 3      | 约束     | “是否存在性能、兼容性或时间线约束？”                |
| 4      | 非目标   | “我们明确**不**应该构建什么？”                      |
| 5      | 现有模式 | “我们应该遵循 X 中使用的模式，还是需要采用新方法？” |
| 6      | 边界情况 | “当出现 [边界条件] 时，应该发生什么？”              |

### 提问规则

| 规则             | 理由                               |
| ---------------- | ---------------------------------- |
| 每条消息一个问题 | 防止认知超载                       |
| 优先使用选择题   | 回答更快，减少歧义                 |
| 提问前先调研     | 尊重用户时间 —— 能自己发现的自己查 |
| 可测试的标准     | 模糊的回答会导致模糊的实现         |

**暂停 —— 在以下条件满足前，切勿进入阶段 3：**

- [ ] 你已理解目的和成功标准
- [ ] 你已识别约束和非目标
- [ ] 不存在关键性歧义

---

## 阶段 3：方案设计

**目标：** 提出 2-3 个具体方案，包含权衡分析并给出明确建议。

每个方案需包含：

| 部分       | 内容                   |
| ---------- | ---------------------- |
| 架构摘要   | 2-3 句话描述该方案     |
| 关键文件   | 需创建/修改的准确路径  |
| 依赖项     | 外部依赖或破坏性变更   |
| 权衡       | 明确的优缺点           |
| 工作量估算 | 任务数量（而非小时数） |
| 风险等级   | 低 / 中 / 高，附解释   |

### 方案选择决策表

| 因素               | 权重 | 评估方法                     |
| ------------------ | ---- | ---------------------------- |
| 与现有模式的一致性 | 高   | 是否符合当前代码库约定？     |
| 简洁性             | 高   | 满足需求的组件/环节最少      |
| 可测试性           | 中   | 每个组件能否独立测试？       |
| 未来可扩展性       | 低   | 仅当用户提及未来计划时才考虑 |
| 性能               | 可变 | 仅当用户指定性能约束时才考虑 |

**首先给出你推荐的方案。** 解释为何在给定约束下它是最佳选择。提供替代方案以表明你已全面权衡过各种可能性。

**暂停 —— 在以下条件满足前，切勿进入阶段 4：**

- [ ] 你已提出至少 2 个方案
- [ ] 每个方案均已记录权衡分析
- [ ] 你已给出明确的推荐并附推理

---

## 阶段 4：计划文档

**目标：** 编写详细、可执行的计划文档并获得明确批准。

### 计划文档格式

```markdown
# [功能名称] 实施计划

**目标：** [一句话描述]
**架构：** [2-3 句话]
**方案：** [选择哪个方案及原因]
**规范参考：** [规范文件路径，例如 `docs/specs/YYYY-MM-DD-<topic>/`]

---

### 任务 N: [组件名称]

**规范 AC：** [此任务满足规范中的哪项验收标准]

**文件：**

- 创建: `exact/path/to/file.ext`
- 修改: `exact/path/to/existing.ext`
- 测试: `tests/exact/path/to/test.ext`

**步骤：**

1. 编写失败的测试（与上述规范验收标准对齐）
2. 运行测试以验证其失败
3. 编写最小化实现代码
4. 运行测试以验证其通过
5. 提交代码

**验证：** [验证此任务的准确命令]
```

### 计划质量检查清单

| 标准                       | 检查项                                        |
| -------------------------- | --------------------------------------------- |
| 每个任务都有准确的文件路径 | 禁止使用“src/ 下的某处”                       |
| 每个任务都有验证命令       | 禁止使用“肉眼检查”                            |
| 任务在适用时引用规范 AC    | 禁止出现脱离规范锚点的孤立任务                |
| 任务按依赖关系排序         | 禁止向前引用                                  |
| 每个任务耗时 2-5 分钟      | 禁止“实现整个模块”                            |
| TDD 步骤明确               | 每个任务遵循 RED-GREEN-REFACTOR（红-绿-重构） |

将计划保存至 `docs/plans/<date>_<topic>/plan.md`。

**暂停 —— 在以下条件满足前，切勿进入阶段 5：**

- [ ] 计划文档已编写并保存
- [ ] 每个任务均包含文件路径、步骤和验证方法
- [ ] 用户已明确批准计划（回复“是”、“批准”、“继续”等）

---

## 阶段 5：过渡到执行

**目标：** 将已批准的计划移交至相应的执行技能。

### 过渡决策表

| 场景                          | 下一个技能                    | 理由                 |
| ----------------------------- | ----------------------------- | -------------------- |
| 标准实施（少于 10 个任务）    | `task-management`             | 顺序跟踪执行         |
| 大型实施（10 个以上独立任务） | `subagent-driven-development` | 并行执行，带审查关卡 |
| 自主开发会话                  | `autonomous-loop`             | Ralph 风格的迭代执行 |
| 单一聚焦任务                  | `executing-plans`             | 直接执行计划         |

调用所选技能并传入计划文档路径。

## 反模式 / 常见错误

| 反模式                 | 失败原因                       | 正确做法                   |
| ---------------------- | ------------------------------ | -------------------------- |
| “这太简单了不需要规划” | 简单任务也有未经验证的假设     | 照常规划 —— 计划可以很短   |
| “我已经知道方案了”     | 你的方案可能与项目模式冲突     | 记录下来并获取批准         |
| “用户想要快点”         | 糟糕的代码比规划好的代码更耗时 | 规划能防止返工             |
| “这只是个 Bug 修复”    | Bug 修复需要根本原因分析       | 规划修复方案，而非仅打补丁 |
| “我边做边规划”         | 那是即兴发挥，不是规划         | 先规划，后执行             |
| 一次性问 5 个问题      | 让用户不知所措，得到模糊回答   | 每条消息只问一个问题       |
| 只提出 1 个方案        | 缺乏权衡分析，可能错过更优选项 | 始终提出 2-3 个方案        |
| 模糊的文件引用         | “更新测试” —— 哪个测试？       | 始终使用准确文件路径       |
| 耗时 30 分钟以上的任务 | 太大，难以跟踪和验证           | 拆分为 2-5 分钟的任务      |
| 未经批准就开始写代码   | 若方向变更将浪费精力           | 等待明确的“同意”           |

## 反自我辩解防御机制

<HARD-GATE>
在以下条件满足前，**切勿**编写任何代码、创建任何文件或采取任何实施行动：
1. 你已提出澄清问题并理解了需求
2. 你已提出带有权衡分析的方案
3. 用户已明确批准该计划

此规则适用于**每一项**任务，无论其看起来多么简单。
</HARD-GATE>

**铁律：未经批准的计划，绝不写代码。** 没有例外。没有“就做这一件小事”。没有“这很明显”。

如果你脑海中出现以下任何想法，**立即停止**：

- “我就快速搞一下……” —— 不行。先规划。
- “这不需要完整的计划……” —— 需要。计划可以很简短。
- “我之后再补文档……” —— 不行。先写文档。

## 子代理调度机会

| 任务模式                   | 调度目标                                                          | 时机                         |
| -------------------------- | ----------------------------------------------------------------- | ---------------------------- |
| 规划期间的独立研究任务     | `Agent` 工具，`subagent_type="Explore"`                           | 从代码库多个区域收集上下文时 |
| 跨架构层的计划验证         | `Agent` 工具调度 `planner` 代理                                   | 当计划覆盖多个系统边界时     |
| 计划批准后，独立的实施任务 | `Agent` 工具（多个并行，遵循 `dispatching-parallel-agents` 技能） | 当计划步骤之间无依赖关系时   |

调度时请遵循 `dispatching-parallel-agents` 技能协议。

| 技能                             | 关系                           | 时机                     |
| -------------------------------- | ------------------------------ | ------------------------ |
| `brainstorming`                  | 上游 —— 提供设计上下文         | 规划在头脑风暴之后进行   |

## 具体示例

### 示例：小型 Bug 修复计划

```markdown
# 修复：登录按钮禁用状态未清除

**目标：** 修复登录失败后登录按钮仍保持禁用的问题
**架构：** LoginForm 组件中的状态管理 Bug
**方案：** 在 handleSubmit 的 catch 块中重置 `isSubmitting` 状态

### 任务 1：编写失败的测试

**文件：** 测试: `tests/components/LoginForm.test.tsx`
**步骤：** 编写测试，提交无效凭据并验证按钮重新启用
**验证：** `npm test -- --grep "re-enables button after failed login"`

### 任务 2：修复 Bug

**文件：** 修改: `src/components/LoginForm.tsx`
**步骤：** 在 handleSubmit 的 catch 块中添加 `setIsSubmitting(false)`
**验证：** `npm test -- --grep "LoginForm"` —— 全部通过
```

### 示例：过渡命令

计划批准后：

```
计划已批准并保存至 docs/plans/2026-03-15_login-fix/plan.md。
正在调用 task-management 技能以开始跟踪执行。
```

## 验证关卡

在声明计划完成前，请验证：

1. **识别 (IDENTIFY)：** `docs/plans/` 中存在计划文档
2. **运行 (RUN)：** 根据质量检查清单审查计划的完整性
3. **阅读 (READ)：** 验证所有章节均已填写具体细节
4. **规范核对 (SPEC-CHECK)：** 如果 `docs/specs/` 中存在规范，验证每个任务至少引用了一项验收标准
5. **验证 (VERIFY)：** 用户已明确批准
6. **声明 (CLAIM)：** 仅在满足以上条件后，才能过渡到实施阶段

## 核心原则

- **DRY** —— 不要重复自己
- **YAGNI** —— 不要构建当前不需要的东西
- **TDD** —— 适用时先写测试
- **频繁提交** —— 每个任务完成后进行小型、原子化提交
- **准确路径** —— 计划中始终指定准确的文件路径

---

## 技能类型

**严格 (RIGID)** —— 严格遵循此流程执行每项实施任务。各阶段按顺序进行且不可妥协。未经批准的计划，绝不编写代码。
